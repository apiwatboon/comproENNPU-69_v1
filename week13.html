Project Brief

📋 กติกาโปรเจกต์

🎯

เป้าหมาย

โปรแกรมภาษาซีที่แก้ปัญหาจริงของสาขาตัวเอง 1 ระบบ ทำงานผ่านเมนูข้อความ (รับคำสั่ง–ทำงาน–วนกลับมาถามใหม่) คอมไพล์ผ่านและใช้งานได้จริงตั้งแต่ต้นจนจบโดยไม่พัง

🧰

ต้องใช้ครบ 5 อย่าง

① ฟังก์ชันอย่างน้อย 4 ตัว · ② เงื่อนไข if–else if หรือ switch · ③ ลูป for/while/do-while · ④ อาร์เรย์เก็บข้อมูล · ⑤ อ่านหรือเขียนไฟล์/CSV อย่างน้อย 1 จุด — ครบ = ใช้ทุกบทที่เรียนมา

📦

สิ่งที่ต้องส่ง

โค้ด .c พร้อมคอมเมนต์ · ไฟล์ข้อมูลตัวอย่าง (.csv/.txt) · ผังงานของฟังก์ชันหลัก 1 ผัง (ทักษะบทที่ 3!) · สไลด์นำเสนอ 5–8 หน้า — ส่งใน LMS ก่อนคาบสัปดาห์ที่ 12

👥

แบ่งบทบาทในทีมให้ชัด

หัวหน้าทีม คุมแผน รวมโค้ด ตัดสินใจ · นักพัฒนา เขียนฟังก์ชันตามที่แบ่งกัน (แบ่งเป็นฟังก์ชันได้เพราะบทที่ 9!) · นักทดสอบ+เอกสาร ลองป้อนข้อมูลแปลก ๆ ให้โปรแกรมพัง แล้วรายงานให้แก้ พร้อมทำสไลด์ — ทุกคนต้องอธิบายโค้ดทุกส่วนได้ตอนถาม-ตอบ

Choose Your Track

🛤️ โจทย์ตามสาขา — เลือก 1 จาก 9 (หรือเสนอเองได้)

ทุกโจทย์ออกแบบให้ทำได้ด้วยความรู้บทที่ 1–12 พอดี — โจทย์เสนอเองต้องผ่านความเห็นชอบจากอาจารย์ภายในคาบนี้

💻

วิศวกรรมคอมพิวเตอร์

C1. ระบบจัดการคะแนนนักศึกษา: บันทึก/แก้ไขคะแนนลง CSV ตัดเกรดอัตโนมัติ สรุปสถิติห้อง

C2. ระบบยืม–คืนอุปกรณ์แล็บ: เช็คของว่าง บันทึกผู้ยืม เตือนเกินกำหนด

C3. เกมทายคำศัพท์คอมพิวเตอร์: คลังคำในไฟล์ จับเวลานับคะแนน เก็บสถิติผู้เล่น

🚚

วิศวกรรมโลจิสติกส์

L1. ระบบคลังสินค้า: รับเข้า–จ่ายออก แจ้งเตือนสต๊อกต่ำ รายงานมูลค่าคงคลังจาก CSV

L2. เครื่องคิดค่าส่งพัสดุ: อัตราขั้นบันไดตามน้ำหนัก+ปลายทาง ออกใบเสร็จเป็นไฟล์

L3. วางแผนบรรทุกรถ: อ่านรายการพัสดุ จัดลงรถตามพิกัดน้ำหนัก สรุปจำนวนเที่ยว

🏭

วิศวกรรมอุตสาหการ

I1. ระบบบันทึกผล QC: เก็บค่าวัดชิ้นงาน เทียบพิกัดความเผื่อ สรุปร้อยละของเสียรายวัน

I2. เครื่องคำนวณ OEE: อ่าน log เครื่องจักรจาก CSV คำนวณ Availability × Performance × Quality

I3. ระบบเบิกวัตถุดิบเข้าไลน์ผลิต: ตัดสต๊อกตามสูตรการผลิต เตือนเมื่อไม่พอ

Recommended Skeleton

🏗️ โครงสร้างโค้ดแนะนำ — ทุกโจทย์ใช้โครงเดียวกัน

สังเกตว่าทุกระบบในชีวิตจริงคือ "ลูปเมนู" ห่อฟังก์ชันย่อย — คัดลอกโครงนี้ไปเริ่มได้เลย แล้วค่อย ๆ เติมเนื้อในทีละฟังก์ชัน ทดสอบทีละชิ้น

project_skeleton.c
#include <stdio.h>
#include <string.h>

int load_data(char names[][30], int qty[]) {   // อ่าน CSV เข้าอาร์เรย์ คืนจำนวนแถว (บท 12)
    ...
}

void add_item(char names[][30], int qty[], int *n) {   // เพิ่มรายการใหม่ (บท 10)
    ...
}

void show_report(char names[][30], int qty[], int n) {   // สรุปสถิติ + จัดคอลัมน์ (บท 5, 11)
    ...
}

void save_data(char names[][30], int qty[], int n) {   // เขียนกลับลงไฟล์ (บท 12)
    ...
}

int main() {
    char names[100][30];
    int qty[100];
    int n = load_data(names, qty);
    int choice;
    do {                                       // ลูปเมนูด้วย do-while (บท 7)
        printf("1=ดูข้อมูล 2=เพิ่ม 3=รายงาน 0=ออก : ");
        scanf("%d", &choice);
        switch (choice) {                      // เมนู = switch (บท 6)
            case 2: add_item(names, qty, &n); break;
            case 3: show_report(names, qty, n); break;
        }
    } while (choice != 0);
    save_data(names, qty, n);                  // บันทึกก่อนจบ
    return 0;
}
🪜

ทำทีละขั้น อย่าทำทีเดียวหมด

เริ่มจากเมนูเปล่า ๆ ที่วนได้ → เติม load_data → เติมทีละเมนู — โปรแกรมต้อง "รันได้เสมอ" ทุกสิ้นวัน อย่าเขียน 200 บรรทัดแล้วค่อยรันครั้งแรก

🧪

ทดสอบแบบคนใจร้าย

ป้อนตัวหนังสือตอนที่ขอตัวเลข เลือกเมนูที่ไม่มี ไฟล์หาย — โปรแกรมที่ดีต้องไม่พัง (ทักษะบทที่ 12: ข้าม + แจ้งเตือน)

💬

คอมเมนต์ = คะแนน

หัวฟังก์ชันทุกตัวเขียนหนึ่งบรรทัด: รับอะไร คืนอะไร — กรรมการอ่านโค้ดไม่ออกภายใน 3 นาที = เสียคะแนนคุณภาพโค้ดแน่นอน

Two-Week Plan

🗓️ แผนงาน 2 สัปดาห์ — ติ๊กตามความคืบหน้าของกลุ่ม

รายการถูกจำไว้ในเครื่อง ใช้เป็นกระดานติดตามของกลุ่มได้เลย — ถ้าจบคาบนี้แล้วยังไม่ผ่าน 4 ข้อแรก รีบปรึกษาอาจารย์ด่วน

คืบหน้า 0 จาก 10 ขั้น
Grading Rubric

📏 เกณฑ์ให้คะแนน (100 คะแนน)

อ่านก่อนเขียนโค้ด — รู้ว่ากรรมการดูอะไร จะได้ลงแรงถูกที่

หัวข้อน้ำหนักได้คะแนนเต็มเมื่อ...
ความถูกต้องของโปรแกรม35ทุกเมนูทำงานถูกต้อง รับมือข้อมูลผิดได้ไม่พัง ข้อมูลบันทึก–อ่านกลับครบถ้วน
การใช้แนวคิดครบถ้วน25ใช้ครบ 5 อย่างตามกติกา และใช้ "ถูกที่" — ฟังก์ชันแบ่งงานสมเหตุสมผล ไม่ใช่แปะให้ครบเฉย ๆ
คุณภาพโค้ด15ตั้งชื่อสื่อความหมาย คอมเมนต์หัวฟังก์ชันครบ ไม่มีโค้ดซ้ำซ้อนที่ควรเป็นฟังก์ชัน
การนำเสนอ + ถามตอบ15เดโมสด 8 นาทีราบรื่น อธิบายเหตุผลการออกแบบได้ ทุกคนตอบคำถามส่วนของตัวเองได้
เอกสารประกอบ10ผังงานตรงกับโค้ดจริง สไลด์ครบ ปัญหา–วิธีแก้–สิ่งที่ได้เรียนรู้ชัดเจน
